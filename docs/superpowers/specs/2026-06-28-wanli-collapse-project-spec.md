# 《万历：山河崩塌》项目方案 SPEC

> 文档类型：项目规格说明书（SPEC）  
> 基线来源：`万历_山河崩塌_项目设计方案.md`  
> 编写日期：2026-06-28  
> 项目形态：单机 Web 历史地图推演游戏  
> 推荐技术栈：React + TypeScript + Vite + Zustand + Web Worker + IndexedDB  
> MVP 时间范围：1573年1月-1621年12月  
> 完整设计时间范围：1573年1月-1662年12月

---

## 1. 项目定义

《万历：山河崩塌》是一款以晚明到明清鼎革为背景的单机 Web 历史地图推演游戏。玩家选择一个历史势力，通过极少量但长期有效的战略决策，影响人口、粮食、财政、战争、叛乱和政权更替的月度演算结果。

项目的第一目标不是还原一个复杂大战略游戏，而是验证一种低操作量历史推演体验：玩家每次主要决定一个军事方向和一个内政重点，系统负责在后台完成月度结算、AI 行为、战争控制权变化、历史事件触发和地图反馈。

### 1.1 一句话规格

玩家用“军事方向 + 内政重点”影响自动月度推演，在地图颜色、事件链和势力兴衰中观察并改变 1573-1621 年晚明秩序的走向。

### 1.2 目标用户

- 对明清史、中国历史、架空历史有兴趣的玩家。
- 喜欢地图涂色、势力变化和长期推演的策略玩家。
- 希望减少微操、专注战略取舍的轻中度玩家。
- 希望通过浏览器直接游玩和存档的用户。

### 1.3 核心体验目标

首局玩家应在 5 分钟内理解：

- 地图颜色代表势力控制。
- 顶部指标代表当前势力状态。
- 左侧决策区可修改军事方向和内政重点。
- 时间推进会按月结算经济、人口、军事、事件和地图变化。
- 事件选项会产生短期可见效果和长期系统影响。

## 2. 设计原则

1. 操作少，但每次决策必须有长期后果。
2. 推演系统可以复杂，前台解释必须直观。
3. 历史事件提供压力和方向，不强制复制真实历史结局。
4. 地图控制权变化是最重要的视觉反馈。
5. 玩家与 AI 使用同一套基础规则，不提供隐藏保护。
6. 所有随机结果必须由固定种子驱动，便于复现、测试和回放。
7. MVP 先验证推演趣味和数值稳定，再扩展完整历史内容。

---

## 3. MVP 范围

### 3.1 MVP 目标

MVP 用于验证以下问题：

- 月度自动推演是否能形成可读、可复现、有变化的历史过程。
- 玩家仅通过两个核心决策是否能明显影响势力走向。
- 地图涂色、月度日志和事件弹窗是否能解释国家兴衰。
- 人口、粮食、财政、战争和叛乱是否形成可理解的因果链。
- AI 势力是否能产生基本合理的扩张、防御和衰退。

### 3.2 MVP 内容

| 模块 | MVP 规格 |
|---|---|
| 时间范围 | 1573年1月-1621年12月 |
| 地图规模 | 12-24 个战略区 |
| 玩家可选势力 | 大明、土默特、建州女真 |
| AI 势力 | 3-5 个，至少包含蒙古/女真/地方势力类型 |
| 决策系统 | 1 个军事方向、1 个内政重点、3 种军事姿态 |
| 内政重点 | 首版实现 6 项：劝课农桑、整顿财政、整军备战、澄清吏治、休养生息、经略边疆 |
| 月度推演 | 人口、粮食、税收、国库、军队、稳定度、控制度、AI 决策 |
| 战争系统 | 区域驻军 + 战线兵力池，不做单支部队微操 |
| 事件系统 | 20-30 个事件，覆盖改革、三大征、建州成长、辽东危机 |
| 叛乱系统 | 基础动态叛乱，支持从低稳定地区生成地方叛乱 |
| 存档 | 手动存档、自动存档、事件前存档、固定种子 |
| 结算 | 简单阶段结算和势力评分 |

### 3.3 MVP 不做

- 武将个人养成。
- 婚姻、宗族和完整宫廷系统。
- 复杂外交谈判。
- 城市建筑和科技树。
- 装备生产与军种细分养成。
- 单支部队移动微操。
- 海战细节。
- 多人联机。
- 移动端深度适配。
- 3D 地图。
- 复杂动画和完整音频系统。

## 4. 核心玩法规格

### 4.1 月度循环

每次推进一个月时，系统按固定顺序处理：

1. 读取当前 `GameState` 和玩家持续生效的决策。
2. 为所有 AI 势力生成军事方向和内政重点。
3. 结算人口、灾害、粮食、税收、国库和军队维护。
4. 结算征募、补给、战线调动和战斗。
5. 结算区域控制度、稳定度和叛乱风险。
6. 检测历史事件、条件事件和势力生成。
7. 写入月度日志、战报、经济报告和历史记录。
8. 更新地图、统计面板、警报和自动暂停状态。

### 4.2 时间控制

必须支持：

- 暂停。
- 单月推进。
- 自动推进。
- 速度档位：1 个月/秒、3 个月/秒、12 个月/秒。
- 自动推进至下一个重大事件。

自动暂停触发条件：

- 重大事件或历史事件触发。
- 玩家首都或重要核心区被攻击。
- 玩家重要区域失守。
- 国库或粮食低于警戒线。
- 全国性叛乱或新政权建立。
- 玩家势力灭亡、继承或达成阶段结算条件。

## 5. 玩家决策系统

### 5.1 军事方向

玩家同一时间只能设置一个主要军事目标。目标可来自：

- 接壤敌对战略区。
- 有军事通道连接的远程区域。
- 正在叛乱的本方区域。
- 被敌方占领的本方核心区域。
- 需要防御的边疆或首都周边区域。

军事方向持续生效，直到玩家修改、目标无效、目标不再接壤、势力继承、重大事件要求重新决策，或系统因首都/核心区危机临时转入防御。

### 5.2 军事姿态

军事姿态作为军事方向的附属参数，不增加额外战略维度。

| 姿态 | 投入 | 风险 | 主要用途 |
|---|---|---|---|
| 保守 | 低 | 低 | 防守、牵制、资源不足时维持战线 |
| 均衡 | 中 | 中 | 常规扩张和稳定战争 |
| 激进 | 高 | 高 | 决战、抢占节点、短期突破 |

### 5.3 内政重点

玩家同一时间只能选择一个内政重点。MVP 首版实现 6 项：

| 内政重点 | 主要收益 | 主要代价 |
|---|---|---|
| 劝课农桑 | 提高农业恢复和粮食产量 | 商业与军备投入减少 |
| 整顿财政 | 提高税收征收效率 | 稳定度和民生压力变差 |
| 整军备战 | 提高征兵、训练和补员 | 消耗国库与粮食 |
| 澄清吏治 | 降低腐败，提高长期行政效率 | 短期税收降低，官僚阻力上升 |
| 休养生息 | 提高人口增长和稳定度 | 扩张和军事准备变慢 |
| 经略边疆 | 提高边疆控制度和补给能力 | 内地发展速度下降 |

后续版本再扩展开放商贸、赈济灾民、强化中央、笼络地方。

### 5.4 决策反馈

每个决策必须在 UI 中显示：

- 当前选择。
- 预计主要收益。
- 预计主要代价。
- 生效范围。
- 最近 6 个月相关指标趋势。
- 若系统临时覆盖军事方向，必须在日志和警报中解释原因。

## 6. 地图系统规格

### 6.1 战略区

MVP 使用 12-24 个战略区。战略区不是现代省份复刻，而是服务于推演节奏、战争通道和历史可读性。

每个区域至少包含：

```ts
interface RegionState {
  id: string;
  name: string;
  terrain: TerrainType;
  climate: ClimateType;
  ownerFactionId: string;
  controllerFactionId: string;
  population: number;
  agriculture: number;
  commerce: number;
  taxCapacity: number;
  stability: number;
  control: number;
  fortification: number;
  grainStock: number;
  garrison: number;
  coreFactionIds: string[];
  connections: string[];
  activeDisasters: string[];
}
```

### 6.2 地图表现

必须支持：

- 按控制者涂色。
- 控制度低时降低饱和度或叠加弱控制表现。
- 战争争夺区显示边框或斜纹。
- 叛乱、缺粮、灾害显示可识别图标。
- 玩家军事方向显示箭头。
- 首都显示特殊标记。
- 悬停显示摘要。
- 点击打开区域详情。

### 6.3 地图信息层

MVP 至少支持：

- 势力控制。
- 人口密度。
- 粮食产量。
- 税收能力。
- 稳定度。
- 军事力量。
- 控制度。

灾害风险、历史核心区、交通与补给线可在 MVP 后扩展。

## 7. 势力系统规格

### 7.1 势力类型

系统必须支持以下势力类别：

- 王朝中央政权。
- 部族联盟。
- 地方土司。
- 农民军。
- 地方军阀。
- 宗室继承政权。
- 海上政权。

MVP 可先实现王朝中央政权、部族联盟、东北部族三类。

### 7.2 势力状态

```ts
interface FactionState {
  id: string;
  name: string;
  type: FactionType;
  treasury: number;
  grainReserve: number;
  armyTotal: number;
  administration: number;
  militaryOrganization: number;
  legitimacy: number;
  corruption: number;
  centralization: number;
  warExhaustion: number;
  capitalRegionId: string;
  primaryColor: string;
  traits: string[];
  aiProfile: AiProfile;
  status: "active" | "collapsed" | "exiled" | "successor" | "observer";
}
```

### 7.3 初始势力

MVP 玩家可选：

- 大明：国力强、领土广、行政负担重。
- 土默特：骑兵与贸易较强，人口和粮食有限。
- 建州女真：初始弱小，成长潜力高。

MVP AI 至少包含：

- 察哈尔或其他蒙古势力。
- 海西女真。
- 播州/西南地方势力或等价地方势力。

### 7.4 势力特性

每个势力必须拥有 2-4 个特性。特性作为数据配置，不写死在 UI 中。

示例：

- 大明：庞大官僚体系、两京制度、边军体系、天下正统。
- 建州女真：八旗雏形、部族整合、低成本动员、辽东目标。

## 8. 推演系统规格

### 8.1 人口

人口每月变化由自然增长、饥荒、战争、疫病、外迁和流入共同决定。

系统必须防止人口无限增长，区域需要人口承载上限。饥荒、战争和低稳定度应显著降低增长，并可能制造流民。

### 8.2 粮食

粮食产量由人口劳动力、农业能力、季节、气候、稳定度、内政重点和灾害修正决定。

粮食消耗包括：

- 人口消费。
- 军队维护。
- 战争补给。
- 赈灾。
- 事件支出。

当粮食低于警戒线时，系统必须生成警报并提高稳定度下降、人口损失和叛乱风险。

### 8.3 财政

税收由人口税基、地区税收能力、控制度、行政效率、商业修正、腐败、战乱损耗决定。

支出至少包含：

- 官僚维护。
- 常备军维护。
- 战争额外支出。
- 赈灾和事件支出。

财政赤字可以短期容忍，但连续赤字应导致债务、筹饷、稳定下降或军队维护不足。

### 8.4 军事

MVP 不创建可拖动棋子，使用区域驻军和战线兵力池。

战斗力由以下因素决定：

- 投入兵力。
- 训练度。
- 士气。
- 组织度。
- 补给。
- 地形。
- 季节。
- 指挥抽象值。
- 随机波动。

战斗结果必须影响：

- 双方兵力。
- 士气。
- 区域控制度。
- 战争疲劳。
- 粮食与国库。
- 月度日志和战报。

### 8.5 控制度与占领

占领不等于完全控制。控制度由驻军、行政投入、合法性、地区认同、敌对核心、战争破坏和叛乱活动共同影响。

低控制度区域应产生：

- 税收折损。
- 粮食调运困难。
- 叛乱风险上升。
- 驻军持续损耗。
- 敌对复占机会。

### 8.6 叛乱与动态势力

叛乱风险由低稳定、流民、税负、灾害、地方武装化、敌对核心影响、驻军压制、赈济和合法性共同决定。

MVP 叛乱阶段：

1. 民变。
2. 地方叛军。
3. 跨区域军事集团。
4. 正式势力。

动态势力必须由条件触发，不按固定日期强制生成。历史时间窗口只作为条件之一。

## 9. 事件系统规格

### 9.1 事件类型

MVP 至少支持：

- 固定时间窗口事件。
- 条件事件。
- 势力专属事件。
- 区域事件。
- 连锁事件。
- 全局事件。

随机事件可在 MVP 后增加。

### 9.2 事件结构

```ts
interface GameEvent {
  id: string;
  name: string;
  category: EventCategory;
  dateWindow?: {
    start: string;
    end: string;
  };
  conditions: EventCondition[];
  targetFactionId?: string;
  targetRegionId?: string;
  description: string;
  options: EventOption[];
  cooldownMonths?: number;
  priority: number;
}
```

每个事件选项必须提供：

- 玩家可理解的短期效果。
- 可选的隐藏长期影响。
- 后续事件标记。
- 对数值、修正、势力、区域或战争状态的影响。

### 9.3 MVP 事件池

MVP 事件应覆盖：

- 张居正改革与去世。
- 清丈田亩、一条鞭法、考成法阻力。
- 宁夏之役、援朝战争、播州之役。
- 军费激增和边军疲惫。
- 建州女真整合。
- 努尔哈赤起兵。
- 后金建立。
- 抚顺失守。
- 萨尔浒之战。
- 辽沈危机。

## 10. AI 系统规格

AI 与玩家使用同一套规则。AI 不能预知未来，只能根据当前状态、历史诉求、势力性格和可见威胁决策。

### 10.1 AI 军事决策

AI 每月评估候选目标，但不必每月修改目标。评分因素：

- 地区战略价值。
- 历史诉求。
- 敌方虚弱度。
- 核心区价值。
- 补给风险。
- 距离。
- 地形成本。
- 本土威胁。
- 财政压力。

### 10.2 AI 内政决策

AI 根据主要危机选择内政重点：

- 粮食不足：劝课农桑。
- 财政赤字：整顿财政。
- 腐败严重：澄清吏治。
- 战争临近：整军备战或经略边疆。
- 稳定度过低：休养生息。
- 控制度下降：经略边疆或强化中央扩展项。

### 10.3 AI 性格

AI 性格必须数据化：

```ts
interface AiProfile {
  aggression: number;
  riskTolerance: number;
  economicFocus: number;
  centralizationPreference: number;
  historicalGoalWeight: number;
  defensePriority: number;
  warEndurance: number;
}
```

## 11. UI 与信息架构

### 11.1 主界面

主界面由四个固定区域组成：

- 顶部信息栏：日期、年号、速度、国库、粮食、人口、军队、稳定度、警报。
- 左侧决策栏：势力概况、军事方向、军事姿态、内政重点、战略建议。
- 中央地图：战略区涂色、图标、箭头、地图层切换。
- 右侧详情栏：区域详情、历史信息、当前影响因素。
- 底部日志区：月度简报、战报、经济、人口、灾害、叛乱、事件、领土变化。

### 11.2 事件弹窗

事件弹窗必须包含：

- 事件标题。
- 时间与地点。
- 简短历史背景。
- 当前局势说明。
- 2-4 个决策选项。
- 每个选项的明确短期效果。
- 确认按钮。

事件弹窗出现时自动暂停。玩家处理完事件后才能继续自动推进。

### 11.3 新手引导

MVP 应提供一个轻量引导流程：

1. 默认推荐选择大明。
2. 指引查看地图和势力概况。
3. 设置军事方向为辽东或等价边疆目标。
4. 设置内政重点为澄清吏治。
5. 推进三个月。
6. 查看税收、粮食和日志变化。
7. 处理一个教学事件。
8. 切换一个地图信息层。
9. 完成一次手动存档。

## 12. 技术架构

### 12.1 模块边界

推演引擎与 UI 必须完全分离。

```text
React UI
  -> Zustand Game Store
  -> Simulation Worker
  -> Simulation Core
  -> Data Files / Random Seed / Save Snapshot
```

### 12.4 数据驱动要求

以下内容必须通过 JSON 或 YAML 配置，不写死在 UI 代码中：

- 势力。
- 区域。
- 事件和事件选项。
- 势力特性。
- 初始数值。
- 胜利目标。
- AI 权重。
- 年号。
- 文本描述。
- 难度修正。

### 12.5 存档

存档必须包含：

- 当前完整状态。
- 随机种子。
- 玩家决策。
- 已触发事件。
- 历史记录。
- 游戏设置。
- 数据版本号。

存档系统必须提供迁移函数，避免数据结构更新后旧存档全部失效。

## 13. 测试与验收

### 13.1 单元测试

必须覆盖：

- 固定种子随机数可复现。
- 日历推进正确。
- 人口、粮食、税收公式边界。
- 战斗力计算不产生 NaN 或负值异常。
- 事件条件检测。
- 存档读写和版本迁移。

### 13.2 批量模拟测试

MVP 阶段必须提供批量模拟脚本，至少支持：

- 运行 100 局 240 个月快速测试。
- 输出势力存续率。
- 输出财政破产次数。
- 输出人口损失范围。
- 输出战争次数和平均持续时间。
- 输出重大事件触发率。

完整版本扩展到 1000 局。

### 13.3 UI 验收

- 新玩家 5 分钟内可完成一次月度推进。
- 顶部信息栏可直接看到关键数据。
- 地图颜色可快速识别势力控制。
- 事件选择不会误触。
- 日志能解释重大数值变化。
- 1080P 屏幕下不遮挡关键操作和指标。

### 13.4 MVP 通过标准

MVP 必须同时满足：

- 1573-1621 的月度推进可连续运行到结束。
- 相同随机种子和相同决策产生相同结果。
- 大明、土默特、建州女真均可完成一局。
- AI 势力能扩张、衰退或被消灭。
- 玩家改变军事方向或内政重点会造成可观察差异。
- 战争会消耗财政、粮食、兵力或稳定度。
- 低稳定地区可能形成叛乱。
- 事件链能触发并影响后续状态。
- 至少一类批量模拟报告可用于调参。

## 14. 分期交付

### 阶段一：推演验证原型

目标：不用真实地图，先验证核心系统是否成立。

范围：

- 6 个虚拟战略区。
- 3 个测试势力。
- 人口、粮食、税收、军队、稳定度、控制度。
- 军事方向和内政重点。
- 自动战争。
- 固定随机种子。
- 运行 240 个月。
- 输出日志和折线统计。

验收：

- 不发生明显数值爆炸。
- 势力能扩张和衰退。
- 战争受补给、经济和稳定影响。
- 玩家决策能改变结果。
- 相同随机种子可复现。

### 阶段二：地图原型

范围：

- SVG 地图接入。
- 12 个历史战略区。
- 区域点击和详情面板。
- 势力涂色。
- 军事方向箭头。
- 基础地图信息层。

验收：

- 地图状态与推演状态同步。
- 区域数据可读。
- 军事目标能从地图选择。

### 阶段三：历史事件原型

范围：

- 1573-1600 事件链。
- 事件数据配置。
- 条件检测。
- 事件弹窗。
- 长期修正。
- 事件日志。

验收：

- 事件按时间窗口和条件触发。
- 事件选项影响后续数值或事件链。
- 自动暂停和存档点正常工作。

### 阶段四：完整 MVP

范围：

- 建州成长和后金建立。

- 20-30 个事件。
- 基础叛乱。
- 完整存档和简单结算。

验收：

- 可从开局完整玩到 1621 年。
- 三个玩家势力均可选。
- 批量模拟报告可用于调整数值。
- MVP 通过标准全部满足。

### 阶段五：完整历史范围

范围：

- 扩展至 1662 年。
- 农民战争。
- 明清鼎革。
- 南明继承。
- 郑氏集团。
- 动态政权生成。

验收：

- 支持 1080 个月完整推演。
- 势力灭亡与继承链可用。
- 结束报告和地图回放可用。

## 15. 风险与控制

### 15.1 数值失控

风险：

- 人口、国库、军队快速膨胀。
- 强势势力滚雪球。
- 一场战争决定全局。
- 历史关键势力无论玩家如何操作都固定崛起或固定失败。

控制：

- 设置人口和军队承载上限。
- 增加占领后的控制成本。
- 增加多战线惩罚。
- 限制远距离补给。
- 引入战争疲劳。
- 使用批量模拟调参。

### 15.2 内容工作量过大

控制：

- 首版只做 1573-1621。
- 地区控制在 12-24 个。
- 事件优先做主线链条。
- 人物不做独立角色系统。
- 事件用条件模板复用。

### 15.3 玩家参与感不足

控制：

- 决策效果必须可见。
- 日志必须解释关键变化。
- 趋势图显示政策前后差异。
- 重大危机自动暂停。
- 事件选择影响后续。
- 地图变化保持高频反馈。

### 15.4 历史与地图表达风险

控制：

- 明确标注项目是历史推演游戏。
- 区分行政归属、实际控制和影响范围。
- 地图用于历史语境，不映射现代政治主张。
- 商业发布前进行历史内容和地图合规审核。

## 16. 成功标准

项目成功不以内容堆叠为标准，而以核心体验是否成立为标准。

MVP 成功标准：

- 玩家只做少量决策，也能明显改变历史走向。
- 玩家能理解经济、战争、灾害、叛乱之间的因果关系。
- 地图颜色变化具有持续观看价值。
- 每局结果不同但可解释。
- 玩家愿意再次开局尝试另一种政策。
- 浏览器运行流畅，自动推演不卡顿。
- 推演核心可扩展，不依赖 UI 重构。

## 18. 明确边界

本 SPEC 不解决完整史料考证、最终地图边界、商业发行合规、音频美术资产制作和完整历史事件文本写作。这些工作应在 MVP 证明核心推演成立后进入独立阶段。

本 SPEC 的当前作用是约束第一版项目建设：先做可运行、可测试、可复现、可解释的历史推演核心，再扩充内容与表现。
